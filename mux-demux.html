The Core Idea

Your computer runs dozens of apps simultaneously ‚Äî a browser, email client, video call, file transfer. All of them share one physical network interface. So how does each app's data reach the right destination, and how does incoming data reach the right app?

The answer is Multiplexing (at the sender) and Demultiplexing (at the receiver), performed by the Transport Layer using port numbers.

Multiplexing

At the sender. Collects data from multiple application sockets, wraps each in a segment with header info (port numbers), and sends them through the network layer.

Many ‚Üí One

Demultiplexing

At the receiver. Examines incoming segments, reads the destination port, and delivers each segment's data to the correct application socket.

One ‚Üí Many

// 04 ‚Äî Port Numbers

How Port Numbers Work

Port numbers are 16-bit integers (0‚Äì65535). They act as the "apartment numbers" in a building ‚Äî the IP address is the building, the port number is the specific unit.

Range Name Examples
0 ‚Äì 1023 Well-Known Ports HTTP:80, HTTPS:443, FTP:21, SMTP:25, DNS:53
1024 ‚Äì 49151 Registered Ports MySQL:3306, PostgreSQL:5432, Redis:6379
49152 ‚Äì 65535 Dynamic / Ephemeral Assigned temporarily to client-side connections
// 05 ‚Äî UDP vs TCP

Connectionless vs Connection-Oriented

UDP and TCP demultiplex differently. UDP uses a 2-tuple; TCP uses a 4-tuple, enabling far more precise socket routing.

UDP

2-Tuple Demux

UDP only checks the destination IP + destination port. Any segment arriving at that port goes to the same socket, regardless of source.
